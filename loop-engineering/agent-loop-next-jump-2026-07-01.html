1. 引言：loop 不是 endgame

当前 loop-engineering skill v0.3.0 已经建立了一个三层架构：

loop-engineering (core)
  ├── loop-engineering-adversarial  →  独立 verifier 审查
  └── loop-engineering-patterns     →  Loop Until Done + Tournament

这六个模式解决了一个核心问题：不要让产出 agent 自己审自己。

但它的本质是 reactive：产出了 → 验证了 → 通过/不通过。它不做三件事：

  1. 跨 run 学习 — verifier 每次都能发现同一个数据源的问题，但从不说"这个问题已经出现 5 次了"
  2. 策略自进化 — rubric 是写死的，不会根据历史数据自动增减
  3. 主动启发人 — verifier 跑完就沉默了

本文提出从 verify 到 observe → learn → coach 的进化路径。

3.2 Toryo — 质量棘轮 + 信任累积

来源：GitHub — JesseRWeigel/toryo（⭐10, MIT） 
Plan → Research → Execute → Review
                              │
                    Score ≥ threshold → git commit (keep)
                    Score < threshold → git revert → Ralph Loop retry
概念做法对 loop 的启发
Quality Ratcheting 只允许质量单向上升——坏结果自动回滚 verifier 不只看单次 pass/fail，看趋势
Trust-Based Delegation agent 通过持续高分积累信任 高信任度的跳过某些 check，低信任度的加强审查
Ralph Loop 失败后把 QA 反馈路由回 agent 重试 verifier 不只是说"你错了"，而是说"错在数据源，换备用源"

3.4 Self-Healing Orchestrators — 失败分类→精准恢复

来源：arXiv:2606.01416 (May 2026)
指标数值
精准恢复成功率98.8%
retry-only 基线94.5%
full replanning 基线93.8%
静默失败（verifier 加入后）0.0%
单次恢复 vs 盲目重试94.0% vs 85.3%
核心教训： 知道"为什么失败"比盲目重试有效得多。当前 verifier 只说"没过"，不说"为什么没过"。

4. 三个进化方向

4.1 方向 A：Meta-Verifier（元验证层）

目标：跨 pipeline 的 pattern discovery

一个独立的 Meta-Verifier 定期跑，不验证单次产出，而是读过去 N 次 verifier 报告，找模式。

输出示例： "过去 7 天日报验证中，'计算准确性' 维度 fail 了 5/7 次，但每次都是矿工 f0xxxx 的 API 返回异常。建议：给这个矿工加 fallback 数据源。"

4.2 方向 B：Adaptive Rubric（自进化检查表）

目标：rubric 不再写死，根据历史数据自动调整

rubric_item.assess():
    if consecutive_pass >= 10:
        → status = "skip" (跳过此检查，省 token)
    elif pass_rate_7d < 0.5:
        → status = "focus" (加严检查，多抽 3 个样本)
    elif is_new_failure_pattern:
        → 自动创建新 rubric_item
    else:
        → status = "normal"

4.3 方向 C：Insight Feed（启发推送）

目标：verifier 主动告诉你"你可能想知道"

⚠️ 日报 anomaly [2026-07-01] 矿工 f0xxxx 的 power 从 3.2PiB 骤降到 1.8PiB
与 7 天均值偏离 >2σ
上次类似事件：2026-06-15（硬件故障，12h 恢复）
→ 建议：检查硬件状态

推送渠道：飞书 DM | Obsidian daily note | 日报 HTML 侧栏角标

4.4 三个方向的协同关系

flowchart TD subgraph "Layer 3: Coach" C[Insight Feed] --> U[用户] end subgraph "Layer 2: Learn" B1[Adaptive Rubric] --> B2[自动增减 check 项] B2 --> B1 end subgraph "Layer 1: Observe" A1[Meta-Verifier] --> A2[跨 run 模式发现] A2 --> A1 end A2 -->|发现新失败模式| B1 B2 -->|显著 pattern| C style A1 fill:#e6f4ea,stroke:#059669 style B1 fill:#fef3c7,stroke:#d97706 style C fill:#e0e7ff,stroke:#4f6ef7
A → B → C 是递进关系。先有 Meta-Verifier 发现模式，模式稳定后升级为 Adaptive Rubric，显著异常推送给用户。不要三个同时搞。

5.1 为什么选 Meta-Verifier 先做

理由说明
不改现有 pipeline现有日报/良率 verifier 完全不动
数据已有verification_report.json 每次验证都写了，只是没人读

7. 差距分析：能做 vs 还不能做

失败模式聚合✅ 能做

delegate_task + JSON 解析，需要统一 report 格式

趋势分析✅ 能做

7 天滑动窗口，阈值需调优

失败分类（区分类型）⚠️ 部分

需要给 verifier 加失败分类字段（ref: Self-Healing Orchestrators）

探索-利用平衡⚠️ 部分

可以手动实现 coverage matrix，需积累数据（ref: Skill Forge）

自动创建新 check 项❌ 不能

自动改 rubric 风险太高，需要人工确认

信任分管理✅ 能做

pass/fail 计数 + 趋势，需定义计算公式（ref: Toryo）

跨 pipeline 关联⚠️ 部分

日报和良率的数据源映射还没做

飞书推送 insight✅ 能做

feishu gateway 已有，Obsidian skill 已有
